# 工作流技能模板 (Workflow Skill Template)

使用此模板定义**多步骤、跨职能的复杂流程**。适用于涉及多个角色、系统集成、条件分支的工作。

## 🎯 流程概览

### 是什么？
一句话描述这个流程的目的。

**示例**：从产品需求提交开始，经过评审、排期、开发、测试，最后上线的完整流程。

### 流程价值
说明这个流程为什么重要以及它带来的价值。

- 📊 **降低风险**：[如何]
- ✅ **保证质量**：[如何]
- 🚀 **加快速度**：[如何时间对比]
- 🎯 **提高一致性**：[如何]

### 关键指标

定义衡量流程是否成功的指标。

| 指标 | 目标值 | 当前值 | 计算方法 |
|-----|--------|--------|---------|
| 人均周期 | X天 | Y天 | (完成日期 - 提交日期) |
| 一次通过率 | X% | Y% | 无返工需求/总数 |
| 参与人数 | 不超过X人 | Y人 | 参与审批的人数 |
| 缺陷率 | <X% | Y% | 上线后缺陷/功能点 |

## 📋 流程阶段详细定义

每个阶段遵循统一格式：

### 📍 阶段 N: [阶段名称]

**阶段目的** (一句话说明这个阶段的目标)

**进入条件** (这个阶段开始前需要满足什么)
- [ ] 条件1
- [ ] 条件2
- [ ] 条件3

**输入** (这个阶段需要的信息或资源)
| 输入项 | 来源 | 格式 | 检查标准 |
|------|------|------|---------|
| 例如 | 哪里来的 | 格式描述 | 质量检查 |

**决策树** (这个阶段的决策逻辑)
```
决策点：[问题]？
├─ 是 → [操作A] → [下一步]
├─ 否 → [操作B] → [下一步]
└─ 其他 → [操作C] → [下一步]
```

**具体步骤** (按顺序)

1. 步骤1
   - ✅ 检查项：[...]
   - 📊 输出: [...]
   - ⏱️ 预计时间: [...]

2. 步骤2
   - ✅ 检查项：[...]
   - 📊 输出: [...]
   - ⏱️ 预计时间: [...]

**退出条件** (这个阶段完成的标志)
- [ ] 条件1
- [ ] 条件2
- [ ] 条件3

**规则和标准** (这个阶段适用的所有规则)
- 规则1: [...]
- 规则2: [...]
- 规则3: [...]

**常见问题和解决方案**
| 问题 | 原因 | 解决方案 | 联系人 |
|-----|-----|---------|--------|
| 例如 | 为什么会这样 | 怎么解决 | 谁来处理 |

**异常处理** (这个阶段的异常情况)
| 异常 | 触发条件 | 处理方式 | 后续步骤 |
|-----|--------|---------|---------|
| 驳回 | [条件] | [怎么做] | [下一步] |
| 延期 | [条件] | [怎么做] | [下一步] |
| 阻塞 | [条件] | [怎么做] | [下一步] |

**示例**：

```
### 📍 阶段 1: 需求评审

**阶段目的**：验证需求的完整性、合理性和优先级

**进入条件**
- [ ] 需求文档已按模板填写
- [ ] 涉及多个部门的需求已沟通对齐
- [ ] 依赖项已明确

**输入**
| 输入项 | 来源 | 格式 | 检查标准 |
|------|------|------|---------|
| 需求文档 | 产品经理 | PRD文档 | 必填字段完整 |
| 相关数据 | 数据部 | 报告/Excel | 数据源清晰 |
| 竞品分析 | 产品经理 | 分析报告 | 包含对标产品 |

**决策树**
```
这个需求能在本季度完成吗？
├─ 否 → 标记为"延期"，加入下季度备选
├─ 可能 → 标记为"待评估"，需要技术评估
└─ 是 → 继续下一步，进行优先级对标
```

**具体步骤**

1. 需求文档完整性检查
   - ✅ 检查项：5个必填字段、3个附件
   - 📊 输出: 完整性检查单
   - ⏱️ 预计时间: 10分钟

2. 需求背景理解
   - ✅ 检查项：能用1分钟解释需求背景
   - 📊 输出: 需求背景描述
   - ⏱️ 预计时间: 15分钟

3. 与产品经理Q&A
   - ✅ 检查项：解答相关疑问
   - 📊 输出: 问题记录
   - ⏱️ 预计时间: 20分钟

4. 评分和评论
   - ✅ 检查项：填写评分、反馈
   - 📊 输出: 评审评论表
   - ⏱️ 预计时间: 10分钟

**退出条件**
- [ ] 完成完整性检查
- [ ] 给出通过/驳回/待改进的判断
- [ ] 填写明确的反馈意见
- [ ] 输出文档已更新

**规则和标准**
- 规则1：PRD必须包含业务背景、目标用户、核心功能、成功标准
- 规则2：大型需求（>40小时估算）必须进行风险评估
- 规则3：评审反馈必须在2个工作日内回复
- 规则4：如有跨部门影响，必须征求相关部门意见

**异常处理**
| 异常 | 触发条件 | 处理方式 | 后续步骤 |
|-----|--------|---------|---------|
| 文档不完整 | 缺少必填字段 | 打回给产品，要求补充 | 重新提交 |
| 紧急需求 | 被标记为P0 | 启动加急流程，24小时评审 | 直接排期 |
| 复杂需求 | 涉及3个以上部门 | 安排专项评审会，邀请所有方 | 会议决议 |
```

## 🔀 条件分支和例外流程

### 主流程分支

如果流程不是简单的线性，使用这一节描述分支逻辑。

```
主流程：[步骤1] → [步骤2] → 决策 → [步骤3] → [步骤4]

分支1（简单版本）:
[步骤1] → [步骤2] → 决策：判断为"简单"
       → [简化步骤3] → [步骤4]
       ⏱️ 周期缩短至原来的60%

分支2（复杂版本）:
[步骤1] → [步骤2] → 决策：判断为"复杂"
       → [详细步骤3a] → [详细步骤3b] 
       → [详细步骤3c] → [步骤4]
       ⏱️ 周期延长至原来的150%

分支3（驳回版本）:
[步骤1] → [步骤2] → 决策：判断为"不符合要求"
       → 通知提交方 → 要求修复 → 重新提交 → [回到步骤1]
```

### 异常流程

描述遇到问题时的处理路径。

| 异常情况 | 触发点 | 处理路径 | 告知人 |
|--------|--------|---------|--------|
| 关键人审批超期 | 超过SLA 2天 | 监管升级给主管 | 相关方 |
| 依赖阻塞 | 上游交付延期 | 评估是否可以继续或暂停 | PM、上游 |
| 需求变更 | 进行中提出重大改变 | 评估影响，决定是否改变或延期 | 所有方 |

## ⏰ 时间计划

### 标准周期

```
阶段 1: 2个工作日
阶段 2: 1个工作日
阶段 3: 5个工作日（并行）
阶段 4: 3个工作日
————————————————
总计: 11个工作日 (约2.5周)
```

### 关键路径

哪些阶段在关键路径上？延迟这些阶段会延迟整体交付。

```
关键路径：阶段A → 阶段B → 阶段D
非关键路径：阶段C（可以与B并行）
           阶段E（可以与D并行）
```

### SLA定义

每个阶段的完成SLA。

| 阶段 | 目标完成时间 | 触发条件 | 超期处理 |
|-----|-------------|--------|---------|
| 阶段1 | 1工作日 | 材料完整的情况下 | 发送催促邮件给审批人 |
| 阶段2 | 2工作日 | 无依赖阻塞 | 升级给经理 |
| 阶段3 | 5工作日 | 团队完整、无重大变更 | 评估是否延期 |

## 📊 质量保证

### 关键检查点

在流程的关键点进行质量检查。

| 检查点 | 检查内容 | 检查人 | 失败处置 |
|--------|---------|--------|---------|
| 需求评审后 | 需求完整性、可行性 | 技术lead | 打回重新提交 |
| 排期后 | 估算合理性、资源充分 | PM | 重新评估 |
| 开发完成后 | 代码质量、单元测试 | 代码审查人 | 返工修复 |
| 上线前 | 功能完整、性能OK | QA & 产品 | 延期上线 |

### 缺陷管理

定义流程中发现的问题如何处理。

```
缺陷发现 → 分类（Critical/Major/Minor）
        → 分配责任人
        → 设定修复SLA
        → 验证修复
        → 更新流程
```

## 💬 与LLM的交互提示

将此流程提交给LLM时：

```
我想让你学会这个完整的工作流程："[流程名称]"

【流程信息】
[复制以上所有内容]

【期望的AI行为】
1. 理解流程的各个阶段和决策规则
2. 当提交新的需求/案例时，自动按流程推进
3. 在关键决策点主动要求人类审批
4. 记录流程执行的元数据（时间、参与者、决策）
5. 自动生成流程状态报告

【自动化范围】
- ✅ 自动完成的：数据收集、分类、通知发送
- ⚠️ 需要人工的：关键决策、审批、异常处理
- 🤖 协助的：生成建议、填充模板、效率检查

【反馈机制】
每个流程周期后，我会：
1. 评估自动化的有效性
2. 提供改进点
3. 更新流程规则
```
