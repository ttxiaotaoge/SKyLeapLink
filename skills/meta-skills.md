# 元技能指导 - Meta Skills Framework

这是一份指导文档，帮助你系统地分析、定义和转化你的工作为AI可执行的技能集。

## 🎯 什么是元技能（Meta Skill）？

**元技能**是关于"如何定义技能"的技能。它是一个方法论框架，指导你：
1. 深入思考你做了什么
2. 系统化地表达工作流程
3. 找到隐藏的规则和决策标准
4. 将隐性知识转化为显性指令

## 🔍 自我审视5问法

在定义任何技能前，先问自己这5个问题：

### 1️⃣ **我通常做什么？**
- **目标** - 这项工作的最终目标是什么？
- **触发条件** - 什么情况下需要执行这项工作？
- **成功标准** - 什么样的结果算是成功的？

**示例：**
```
工作：编写周报
目标：总结本周成果，规划下周任务
触发条件：每周五下午
成功标准：包含3个成果、2个挑战、3个下周计划
```

### 2️⃣ **我如何做的？**
- **步骤** - 按顺序列出关键步骤
- **工具** - 使用了哪些工具或资源？
- **数据来源** - 数据从哪里获取？

**示例：**
```
步骤1：收集本周所有待办任务完成情况
步骤2：总结为3个主要成果
步骤3：分析遇到的问题和障碍
步骤4：规划下周3项优先工作
步骤5：用指定模板格式化输出
```

### 3️⃣ **我的决策规则是什么？**
- **评判标准** - 如何判断什么是好的结果？
- **权衡因素** - 有哪些相互制约的因素？
- **例外情况** - 什么时候不适用这个流程？

**示例：**
```
规则1：按优先级排序（紧急度 > 战略性 > 复杂度）
规则2：成果数不少于3个，不超过5个
规则3：如果遇到突发事件，要特别说明
```

### 4️⃣ **我需要哪些背景知识？**
- **领域知识** - 需要什么专业知识？
- **上下文** - 需要了解什么背景信息？
- **隐性假设** - 我做决定时的潜在假设？

**示例：**
```
知识：公司战略方向
知识：各部门KPI
知识：团队成员能力评估
假设：稳定的业务环境、正常的资源配置
```

### 5️⃣ **我怎样改进和优化？**
- **反馈** - 如何评估执行效果？
- **迭代** - 需要调整什么？
- **学习** - 从经验中学到了什么？

**示例：**
```
反馈：每月评审周报质量
迭代：根据反馈调整成果呈现方式
学习：发现加入"团队合作"视角能提升质量
```

## 📋 技能定义框架

### 层级1：工作分类

首先，将你的工作分类：

```
┌─ 核心专业技能 (Core Professional Skills)
│  └─ 你专业领域的核心工作
│
├─ 跨职能技能 (Cross-functional Skills)
│  └─ 多个部门都需要的能力
│
├─ 流程管理技能 (Process Management Skills)
│  └─ 工作流程和系统
│
└─ 决策支持技能 (Decision Support Skills)
   └─ 帮助决策和判断
```

### 层级2：技能详细度

根据复杂度选择定义深度：

| 深度等级 | 场景 | 定义内容 |
|--------|-----|--------|
| **L1-轻量** | 简单重复任务 | 输入→处理→输出 |
| **L2-标准** | 普通工作流程 | 4-6个步骤+决策规则 |
| **L3-详尽** | 复杂决策过程 | 详步骤+多维规则+异常处理 |
| **L4-企业级** | 关键业务流程 | 完整文档+审计+版本控制 |

### 层级3：技能组合

单个技能可以组合成**工作流**：

```
技能A: 收集信息
    ↓
技能B: 分析数据
    ↓
技能C: 生成报告
    ↓
技能D: 发送通知
```

## 🏗️ 技能建模的4个维度

### 维度1：输入维度
```yaml
输入源:
  - 数据文件 (files, databases)
  - 人工输入 (manual input)
  - 系统接口 (APIs)
  - 日程事件 (calendar events)
```

### 维度2：处理维度
```yaml
处理类型:
  - 信息收集 (gather)
  - 数据分析 (analyze)
  - 内容生成 (generate)
  - 决策判断 (decide)
  - 流程协调 (coordinate)
```

### 维度3：规则维度
```yaml
规则类型:
  - 业务规则 (business rules)
  - 技术规范 (technical standards)
  - 质量标准 (quality criteria)
  - 合规要求 (compliance)
```

### 维度4：输出维度
```yaml
输出形式:
  - 文档报告 (documents)
  - 结构化数据 (data)
  - 可视化展示 (visuals)
  - 自动通知 (notifications)
  - 系统更新 (system updates)
```

## 🎯 定义技能时的检查清单

使用这个清单确保你的技能定义完整和令人信服：

### ✅ 清晰性检查
- [ ] 技能名称清晰易懂，见名知意
- [ ] 能用一句话解释这个技能的目的
- [ ] 成功标准是明确的、可测量的
- [ ] 外人能根据说明独立完成这项工作

### ✅ 完整性检查
- [ ] 列出了所有必要的输入
- [ ] 定义了所有关键决策点
- [ ] 明确了异常处理方法
- [ ] 包含了必要的背景知识

### ✅ 可执行性检查
- [ ] 步骤按逻辑顺序排列
- [ ] 每个步骤都很明确，不模糊
- [ ] 没有需要高度主观判断的内容
- [ ] LLM能理解并执行

### ✅ 可维护性检查
- [ ] 规则是稳定的还是经常变化？
- [ ] 有版本控制和更新机制吗？
- [ ] 异常规则是否另外记录？
- [ ] 是否有人维护这个定义？

## 💬 与LLM的最佳对话方式

### 第1阶段：信息收集

```
你：我有一个关于"定义内容营销策略"的技能想教AI学习
LLM：很好！为了深入理解，请告诉我...
      1. 通常需要多长时间完成？
      2. 成功的标准是什么？
      3. 你在这个过程中做的最复杂的决定是什么？
```

### 第2阶段：规则抽取

```
LLM：根据你的描述，我理解了4个关键步骤。
     让我总结一下每个步骤的规则：
     [LLM总结的规则]
     这些规则准确吗？需要调整吗？
```

### 第3阶段：场景验证

```
LLM：现在让我用这个规则处理一个假设场景...
     [LLM演示如何执行]
     这样做对吗？需要改进什么？
```

### 第4阶段：优化和版本管理

```
LLM：基于我们的讨论，这是最终版本的技能定义。
     [最终定义]
     你想现在部署它还是继续改进？
```

## 🔄 持续改进循环

```
1. 部署 (Deploy)
   └─ 让AI使用这个技能
   
2. 观察 (Observe)
   └─ 记录执行过程中的问题
   
3. 分析 (Analyze)
   └─ 理解失败和成功的原因
   
4. 改进 (Improve)
   └─ 更新规则和步骤
   
5. 验证 (Validate)
   └─ 用新规则再试一遍
   
6. 迭代 (Iterate)
   └─ 回到步骤2
```

## 🎓 高级技巧

### 1. 发现隐性规则
你的大脑中有很多"自动化"的决策规则。要发现它们，问自己：
- "如果...那么..."这样的情况，我会怎么做？
- 我做决定时通常考虑什么因素？
- 有没有某些决定我做起来很快，因为有捷径？

### 2. 处理模糊性
某些工作本质上包含主观判断。处理方法：
- 使用评分标准而不是绝对判断
- 定义评价维度（如：1-10分、低中高等）
- 给出具体例子说明每个级别

### 3. 处理例外和特殊情况
```yaml
通用规则: [标准流程]

例外规则:
  - 当 [条件A] 时: [特殊处理]
  - 当 [条件B] 时: [特殊处理]
  - 当 [都不符合] 时: [默认处理或人工介入]
```

### 4. 处理需要创意的工作
对于需要创意的任务（如写作、设计），不是教AI"做什么"，而是教：
- 参考标准和范例
- 质量评价维度
- 创意约束条件
- 反馈循环

## 📊 示例：完整的技能演进过程

### 第0版：初始想法
```
"写报告"
```

### 第1版：基本分解
```
输入：这周的数据
处理：总结要点
输出：报告
```

### 第2版：详细步骤
```
1. 收集本周所有相关数据
2. 按照X、Y、Z三个维度分析
3. 提取关键洞察（最多5个）
4. 撰写报告
5. 自检质量
```

### 第3版：加入规则
```
收集数据时的规则：
- 只包含已确认的数据
- 舍弃异常值或特殊情况
- 优先级按[标准]排序

分析时的规则：
- 必须包含时间序列对比
- 必须提出可行性建议
- 避免空洞的评论

质量检查：
- 长度800-1200字
- 至少2张图表
- 必须有结论和建议
```
