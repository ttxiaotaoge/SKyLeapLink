这是一个使用基础技能模板的完整示例。你可以复制这个格式来创建自己的技能。

## 🎯 技能概览

### 是什么？
每周五下午，根据本周工作完成情况、遇到的问题和下周计划，生成一份结构化的周报，汇报给直属利益相关方。

### 为什么需要这个技能？

- ⏰ **时间节省**：从原来的60分钟减少到15分钟(75%时间节省)
- 📊 **一致性**：统一格式和呈现，便于汇总和趋势分析
- 📈 **透明度**：高层管理能快速了解进展
- 🎯 **反思总结**：强制进行周期性的工作总结和规划

### 成功标准

✅ 周报包含3-5项主要成果  
✅ 用指定的Markdown模板格式  
✅ 所有数据来自Jira系统（已验证）  
✅ 文字清晰，无语法错误  
✅ 每周五下午5点前完成  
✅ 自动发送给主管和相关方  

---

## 📥 输入

### 输入来源

| 输入项 | 来源 | 格式 | 访问方式 |
|------|------|------|---------|
| 本周任务完成 | Jira系统 | JSON API | 个人项目board |
| 周会议记录 | Confluence | 文本 | 团队wiki链接 |
| 遇到的问题 | Slack #status | 消息历史 | 工作频道 |
| 业绩数据 | 内部数据库 | CSV导出 | 周五早上快照 |

### 访问权限

```
- Jira: "项目成员"以上权限
- Confluence: 可读权限
- Slack: #status频道成员
- 数据库: 账户有访问权限
```

### 数据质量标准

- Jira任务必须已标记为"Done"
- 数据快照时间为本周五上午10点
- 所有任务的"Story Points"和"完成时间"必须填写
- 没有重复的任务记录

---

## ⚙️ 处理过程

### 步骤序列

**步骤 1: 收集数据**
- 操作：从Jira导出本周完成的所有任务，聚合数据
- 使用工具：Jira Dashboard / API脚本
- 输入：
  - Jira访问权限
  - 截止日期：本周五
- 输出：
  ```json
  {
    "total_issues": 12,
    "story_points": 34,
    "categories": {
      "feature": 5,
      "bugfix": 4,
      "refactor": 3
    }
  }
  ```
- ⏱️ 预计时间：10分钟

**步骤 2: 整理关键成果**
- 操作：从完成任务中提取最重要的3-5项，用1-2句话描述其价值
- 使用工具：文字编辑器/AI辅助
- 输入：
  - 完成的任务列表
  - 对"重要度"的判断标准（见决策规则）
- 输出：
  ```
  1. 完成用户认证系统2.0版本，提升登录速度30%
  2. 修复关键支付流程bug，影响50000+用户
  3. 完成API文档更新，便于第三方集成
  ```
- ⏱️ 预计时间：15分钟

**步骤 3: 总结遇到的问题/挑战**
- 操作：盘点本周遇到的阻塞、困难或依赖问题
- 使用工具：Jira / Slack搜索
- 输入：
  - Slack消息历史
  - Jira中标记的"阻塞"状态任务
  - 会议记录
- 输出：
  ```
  - 服务器性能问题（延期了数据库优化）
  - 设计稿返工2次（来自产品反馈不清晰）
  ```
- ⏱️ 预计时间：10分钟

**步骤 4: 规划下周工作**
- 操作：根据优先级和依赖关系，列出下周计划的3项主要任务
- 使用工具：Jira Backlog
- 输入：
  - Product Backlog
  - 优先级判断标准（见决策规则）
- 输出：
  ```
  1. 完成性能优化（已有技术方案）
  2. 启动新功能开发（等待设计稿）
  3. 技术债清理 (低优先级)
  ```
- ⏱️ 预计时间：10分钟

---

## 🧠 决策规则

### 规则1: 重要度判断标准

```
IF 任务影响用户数量 > 10,000 THEN 重要度 = 高
ELSE IF 任务影响关键业务流程 THEN 重要度 = 高
ELSE IF 任务是技术债或内部优化 THEN 重要度 = 低
ELSE 重要度 = 中
```

### 规则2: 下周优先级排序

```
优先级 = (业务紧急度 × 40%) + (用户影响度 × 30%) + (依赖关键性 × 20%) + (技术复杂度 × 10%)

高价值 = 优先级 > 70
中价值 = 优先级 40-70
低价值 = 优先级 < 40
```

### 规则3: 成果描述标准

```
GOOD:  完成用户认证系统2.0版本，提升登录速度30%，影响50万日活用户
BAD:   完成认证系统工作
WHY:   缺少定量的影响说明
```

### 规则4: 数据聚合规则

```
IF 同一个feature中有多个细分任务（子任务） THEN 合并为一项成果
ELSE IF 本周新开但未完成的任务 THEN 不计入本周成果
```

## 🚧 本周挑战与反思

1. **服务器性能问题导致1天延期** 
   - 根本原因：数据库查询未优化，高并发时响应缓慢
   - 对策：已立项进行数据库优化（下周启动）
   - 学习：在设计阶段需要关注性能指标

2. **设计稿返工2次**
   - 根本原因：产品需求表达不清晰，设计理解偏差
   - 对策：后续需求评审中加入visual checkpoints
   - 学习：前期沟通投入能节省后期返工

3. **新API文档编写比预期多花3小时**
   - 根本原因：示例代码较复杂，需要多语言支持
   - 对策：未来考虑使用文档生成工具
   - 学习：文档计划需要buffer
